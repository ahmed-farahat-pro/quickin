# QuickIn iOS App

Native SwiftUI client for QuickIn, living at `mobile/ios/`. SwiftUI + iOS 17, generated by **XcodeGen** (`project.yml` → `QuickIn.xcodeproj`). No storyboard, no AppDelegate-driven UI — pure SwiftUI `App` scene. ~79 Swift files, ~29k LOC, all flat under `Sources/`.

Unlike the older web docs (which mention Supabase), the iOS app talks **only** to the deployed **backend** Next.js project (`quickin-backend.vercel.app`) over plain `URLSession` + `Codable`. No Supabase SDK, no Firebase Auth — auth is the same stateless `qk_token` HMAC bearer the web uses, stored in `UserDefaults`. The only third-party SDKs are **GoogleMaps** (Explore map / host pin-picker) and **FirebaseMessaging** (push tokens only).

## 2. `Config.swift` — backend & build-config awareness

`enum Config` (`Sources/Config.swift`) is the single place URLs/keys live.

- **`apiBaseURL`** (`Config.swift:18-22`) — **build-config aware** via `#if DEBUG / #else`, but **both branches currently point at `https://quickin-backend.vercel.app`** so the Simulator/device work with no local server. The comment notes you switch the DEBUG branch back to `http://127.0.0.1:3000` for local dev against `npm run dev`. **All app networking is `Config.apiBaseURL + path`.**
- **`idOcrBaseURL`** (`Config.swift:58-63`) — a **separate** base for the local Egyptian National ID OCR server on port 8000. **Compile-conditional on `targetEnvironment(simulator)`**: Simulator uses `http://localhost:8000`; a physical device uses `http://<devServerLANIP>:8000`. `devServerLANIP` (`:56`, currently `192.168.8.24`) is the dev Mac's LAN IP — update it when the IP changes (`ipconfig getifaddr en0`) and keep the device on the same Wi-Fi. Plain HTTP works because `NSAllowsLocalNetworking` is set in the Info.plist (see project.yml).
- **`googleClientID`** (`:33`) — iOS OAuth client id; when empty the Google button shows an inline note instead of running OAuth.
- **`googleMapsAPIKey`** (`:45`) — Google Maps iOS SDK key; provided to `GMSServices` at launch in `QuickInApp.init()`.
- **`googleRedirectScheme`** (`:65-73`) — computed reversed-client-id scheme for the OAuth redirect, derived from `googleClientID`.

So there are **two backends in play**: the production Next.js API (`apiBaseURL`) for everything, and a localhost-only OCR server (`idOcrBaseURL`) for ID scanning.

## 3. Auth layer

### Models (`AuthService.swift`)
- **`AuthUser`** (`:11-75`) — `Codable, Equatable`. Fields: `id`, `email`, `fullName` (`full_name`), `provider`, `avatarURL` (`avatar_url`), `role`, `isHost` (`is_host`). Custom `init(from:)` (`:47-60`) makes `isHost` resilient: prefers the explicit boolean, else infers `isHost = (role == "host")` so older responses still read as host. `promotedToHost()` (`:64-74`) returns a copy with `isHost = true` after "become host". **Unified-account contract**: one account per person; `isHost` is the source of truth the UI branches on, `role` is a derived convenience.
- **`AuthSuccess`** (`:78-81`) — the success shape `{ token, user }`.
- **`AuthErrorBody`** (private, `:86-90`) — error shape `{ error }` plus optional `needsVerification: Bool?` and `email: String?` so an unverified login can route to OTP.
- **`PendingBody`** (private, `:94-97`) — `{ pending: true, email }` returned by `/signup` and `/resend-otp` (no token issued until verified).
- **`BecomeHostResponse`** (private, `:102-105`) — `{ ok, user }` from `/api/local/host/become`.

### `AuthOutcome` (`:109-116`)
`enum AuthOutcome: Equatable` with three cases:
- `.authenticated` — session stored, signed in.
- `.needsVerification(email:)` — email needs OTP; present the OTP screen.
- `.failed` — message already on `errorMessage`.

### `AuthStore` (`:122-643`, `@MainActor final class … ObservableObject`)
The session store. Published: `isAuthenticated`, `user`, `errorMessage`, `isLoading`.
- **Persistence**: token in `UserDefaults` key `qk_token` (`AuthStore.tokenKey`, public so other services read it), user JSON under `qk_user`. `restoreSession()` (`:147-156`) runs in `init` to restore on launch. `currentToken` (`:170-173`) exposes the bearer.
- **Public actions** (all `async`, set `isLoading`/`errorMessage`):
  - `login(email:password:)` (`:238`) → `POST /api/auth/login`. On HTTP **403 + `needsVerification == true`** returns `.needsVerification(email:)`; on 2xx persists and returns `.authenticated`.
  - `loginDeferred(...)` (`:187-216`) — like `login` but does **not** publish `isAuthenticated`; returns `(AuthOutcome, AuthSuccess?)` so the sign-in screen can show the "Enable Face ID?" prompt before the presenting sheet auto-dismisses. Committed later via `commitDeferredSession(_:)` (`:221-223`).
  - `signup(name:email:password:country:)` (`:275`) → `POST /api/auth/signup`. Success returns `{ pending: true }` (no token) → `.needsVerification(email:)`. `country` (optional English display name) and the OTP path mean **signup never logs you in directly**.
  - `verifyOTP(email:code:referralCode:)` (`:313`) → `POST /api/auth/verify-otp`; on success persists `{ token, user }` and returns `.authenticated`. Optional `referral_code`.
  - `forgotPassword(email:)` (`:340`) / `resetPassword(email:code:password:)` (`:365`) — `POST /api/auth/forgot-password` then `/reset-password` (resets log you in).
  - `resendOTP(email:)` (`:391`) → `POST /api/auth/resend-otp`.
  - `becomeHost()` (`:417`) → `POST /api/local/host/become` (Bearer token); flips `is_host` on the same account in place, no re-login.
  - `adopt(token:user:)` (`:479`) — persist a session obtained outside the email flow (Apple/Google/biometric).
  - `exchangeSocial(path:body:)` (`:525`) — POST a social id-token to `/api/auth/apple` or `/api/auth/google`, adopt the returned session.
  - `applyProfile(...)` (`:497) / `applyHostUser(...)` (`:464`) — merge edited profile / host flag into cached `AuthUser` (and Keychain copy) without re-login.
  - `logout()` (`:564`) — clears `qk_token`/`qk_user`, **keeps** the biometric Keychain session on purpose, posts `.qkAuthDidLogout` so `WishlistStore` flushes.
- **Networking** (private, `:585-642`): `send(path:body:)` POSTs JSON and classifies the response into a private `SendResult` enum (`.success(Data)` / `.failure(HTTPURLResponse, Data)` / `.transport(String)`). `decodeSession` and `setErrorFromResponse` handle bodies.

### OTP screen (`OTPVerificationView.swift`) + AuthView routing
- **`OTPVerificationView`** (`OTPVerificationView.swift:12`) — takes `email`, optional `referralCode`, and `onVerified` callback. UI is six boxed digits backed by one hidden `TextField` (`keyboardType .numberPad`, `textContentType .oneTimeCode` for autofill). `verify()` (`:201`) calls `auth.verifyOTP(...)`; on `.authenticated` fires `onVerified()`. `resend()` (`:208`) calls `auth.resendOTP` and starts a **30-second cooldown** (`startCooldown()`, `:220`) mirroring the server.
- **`AuthView` routing** — declares a private `OTPSession: Identifiable` struct (`AuthView.swift:29-33`) wrapping `{ email, referralCode }` and drives `@State otpSession` via `.fullScreenCover(item: $otpSession)` (`:111-118`). The pivot is `handle(_:session:)` (`:479-507`):
  - `.authenticated` + non-nil deferred `session` → stage `pendingBiometricSession`, show the enable-biometric sheet.
  - `.needsVerification(email:)` → for an **unverified login** first `await auth.resendOTP(email:)` (signup already emailed a code), then set `otpSession = OTPSession(email:, referralCode:)` (referral carried only on the signup path).
  - `.failed` → no-op (error already shown).
  - `submit()` (`:456`) decides signup vs login, and uses `loginDeferred` only when biometrics are available but no session is stored yet.
- Social: `handleApple(_:)` (`:552`) forwards Apple `identityToken` + name to `exchangeSocial("/api/auth/apple")`; `handleGoogle()` (`:587`) runs `GoogleSignIn.signIn()` then `exchangeSocial("/api/auth/google")`. Biometric unlock: `handleBiometricSignIn()` (`:529`) prompts Face ID, loads the Keychain session, calls `auth.adopt(...)`.

## 5. Networking pattern

Consistent across all `*Service.swift`:
- A `struct` with `static let shared` (e.g. `BookingService.shared`) — value-type, dependency-free.
- A private `URLSession` configured with `timeoutIntervalForRequest = 15` and `waitsForConnectivity = true` (`BookingService.swift:14-19` is the template; `SupabaseService`, `AuthStore` use the same).
- All requests are `Config.apiBaseURL + path`. Headers: `Content-Type: application/json`, `Accept: application/json`, and for authenticated calls `Authorization: Bearer <token>`.
- **Token source**: services read `UserDefaults.standard.string(forKey: AuthStore.tokenKey)` directly (e.g. `BookingService.token`, `:22-25`) so they stay decoupled from `AuthStore`. Guests have `nil` → service throws `notSignedIn`.
- **Bodies/decoding**: `JSONSerialization.data(withJSONObject:)` for requests; `JSONDecoder().decode(Model.self, …)` for responses. DTOs live in `Models.swift` with `snake_case` ↔ camelCase via `CodingKeys`.
- **Error handling**: check `HTTPURLResponse.statusCode`; 2xx decode, 401 → not-signed-in, other 4xx/5xx → decode `{ error }` into a typed error (e.g. `BookingError.message`). `AuthStore` uses its own `SendResult` 3-way enum.
- No retry/interceptor layer; each method is self-contained.

### Endpoints the app calls
**Auth (backend):** `/api/auth/{signup,login,verify-otp,resend-otp,forgot-password,reset-password,apple,google}`.
**Local-stack API:** `/api/local/{listings,services,bookings,wishlist,profile,notifications,reviews,guest-reviews,service-requests,reports,receipts,referrals,regions,currencies,promo/validate,change-password,verification}`, `/api/local/host/{become,listings,bookings,services,earnings,analytics,service-requests}`, `/api/local/ai/{chat,search,listing-description}`, `/api/local/notifications/{register,read-all}`, `/api/local/users/…`.
**OCR (local server):** `Config.idOcrBaseURL + /scan-base64` (`EgyptianIDScanService.swift:80`).

## 7. How it talks to the backend

- **Mobile points at the BACKEND project**, not the frontend web app: `Config.apiBaseURL = https://quickin-backend.vercel.app` (both DEBUG and RELEASE today). The backend is a separate Next.js + Vercel project sharing the **same Neon Postgres** as the frontend.
- **Auth/OTP flow** mirrors the web's local stack: signup creates an **unverified** user and the backend emails a 6-digit OTP via its own SMTP; `users.email_verified` gates login. An unverified login returns **HTTP 403 `{ needsVerification, email }`** → the app shows `OTPVerificationView`. Verify/resend hit `/api/auth/verify-otp` / `/api/auth/resend-otp`. The session is the stateless HMAC **`qk_token`** bearer (sent as `Authorization: Bearer …`).
- **Social sign-in**: the app obtains a provider id-token (Apple `ASAuthorizationAppleIDCredential.identityToken`; Google via ASWebAuthenticationSession + PKCE) and POSTs it to `/api/auth/apple` / `/api/auth/google`, which verify it and return `{ token, user }`.
- **Push**: APNs → FirebaseMessaging mints an FCM token → `NotificationService.registerDeviceToken` POSTs it to `/api/local/notifications/register`; the backend sends push via FCM HTTP v1. All best-effort.
- **Deep links**: shared `https://quickin-frontend.vercel.app/…` Universal Links (AASA on the frontend domain) and `quickin://` scheme both route through `DeepLinkRouter`, which resolves the entity via the same `/api/local/*` endpoints.
- **OCR is the exception**: ID scanning POSTs base64 image data to `Config.idOcrBaseURL` (a **localhost/LAN** server on port 8000), not the Vercel backend.

## 8. Gotchas

- **`SupabaseService.swift` is misnamed.** It is the listings HTTP client and hits `/api/local/listings` — there is **no Supabase** anywhere in the iOS app. Don't let the filename mislead you.
- **`Config.apiBaseURL` DEBUG == RELEASE today.** Both point at production `quickin-backend.vercel.app`. To develop against a local `npm run dev`, manually edit the DEBUG branch back to `http://127.0.0.1:3000` (`Config.swift:18-19`).
- **OCR is a different base URL and a different host.** `idOcrBaseURL` is a localhost/LAN HTTP server on port 8000, compile-switched by `targetEnvironment(simulator)`. On a **physical device** you must update `Config.devServerLANIP` (`Config.swift:56`) to the dev Mac's current LAN IP and be on the same Wi-Fi, and the plain-HTTP call only works because `NSAllowsLocalNetworking` is set.
- **Signing: wrong team blocks device builds.** Default/dev = `U4NBL42U65`; `97DNR5Y3Y5` is TestFlight-distribution only and will fail development signing.
- **Project is generated.** Edit `project.yml` and re-run `xcodegen generate`; do not hand-edit `QuickIn.xcodeproj` (the working tree shows `project.pbxproj` churns).
- **Signup never logs you in.** It returns `{ pending: true }` and forces OTP. Login of an unverified email returns **403** (not a generic failure) — `AuthStore` specifically inspects `needsVerification` to branch, so don't treat 403 as "wrong password".
- **`isHost` is the source of truth, not `role`.** `AuthUser.init(from:)` infers `isHost` from `role == "host"` for legacy payloads; UI branches on `isHost`. Unified account — no host/guest choice at sign-in, and no separate host tab set (host features live under Profile).
- **Tokens are read straight from `UserDefaults` (`qk_token`) by every service**, decoupled from `AuthStore`. If you change the persistence key, update `AuthStore.tokenKey` (services reference it).
- **Logout intentionally keeps the biometric Keychain session** so "Sign in with Face ID" survives logout. It's cleared only by turning Face ID off in Profile → Security, or replaced by a fresh password login. (`AuthStore.logout()`, `AuthService.swift:564`.)
- **Biometric deferred-login dance:** when biometrics are available but no session is stored, sign-in uses `loginDeferred` and holds the session out of `isAuthenticated` until the "Enable Face ID?" sheet is answered (`AuthView.handle`/`finishAuthenticated`). Forgetting to call `commitDeferredSession` would leave the user "authenticated server-side but not signed in" in the UI.
- **OTP resend has a 30s client cooldown** (`OTPVerificationView.startCooldown`) mirroring the server — resending faster is blocked client-side.
- **Firebase pulls a large dependency graph; first `xcodebuild`/resolve is slow.** Only `FirebaseMessaging` is used (push tokens) — there is no Firebase Auth/DB.
