1. Booking with a past check-in returns HTTP 500 instead of 400

Problem. createBooking throws 'Check-in cannot be in the past', but the POST catch-block's status classifier regex /available|after check-in|Invalid|required|maximum guests/i does not match that phrase, so a pure user-input error is returned as a 500. Confirmed live: a past-date booking POST returns 500 {"error":"Check-in cannot be in the past"} while the inverted-date case happens to return 400 only because its message contains 'after check-in'.

Fix. Broaden the regex (or invert the logic) so all createBooking validation messages map to 400. Add 'in the past' and 'after check-in' and 'maximum guests'/'Exceeds the maximum' alternatives, e.g. /not available|in the past|after check-in|Invalid|required|maximum guests|Exceeds/i. Better: have createBooking throw a typed ValidationError and classify on the type rather than string-matching.

2. Public listings API leaks raw Postgres error text via detail: String(err) on 500

Problem. The unauthenticated GET /api/local/listings echoes the stringified driver error to the client on 500. Confirmed live: ?guests=999999999999999999999999 returns {"error":"Failed to load listings","detail":"error: invalid input syntax for type integer: \"1e+24\""}, disclosing the DB column type and driver message. Same detail:String(err) / error:String(err) pattern repeats on listings/[id], bookings, and bookings/[id].

Fix. Stop returning detail/raw error to clients; keep console.error server-side only and return a generic body (e.g. {error:'Failed to load listings'}), optionally gating detail behind NODE_ENV!=='production'. Defense-in-depth: clamp/reject the guests param (Number.isSafeInteger + sane max) in getListings before it reaches Postgres so the overflow never throws.

3. Listing images have no onError fallback and non-image URLs can be persisted

Problem. Card/map/detail <img> tags use the raw stored URL with only a `|| FALLBACK_IMG` guard that fires on a falsy value, never on a present-but-broken/non-image URL; no onError handler exists anywhere in explore. Root cause: the create-listing form accepts arbitrary text as 'Photo URLs' with no image-URL validation, and neither the POST route nor createListing validates them, so a non-image URL (e.g. a google.com/search link) gets stored and rendered as a broken-image icon.

Fix. Add onError={(e)=>{e.currentTarget.onerror=null; e.currentTarget.src=FALLBACK_IMG}} to every card/popup/hero/gallery img. Separately, validate image URLs at create time (require http(s) and a plausible image extension or content-type) in new-listing-form.tsx, the POST /api/local/listings handler, and createListing; backfill/clean any existing listing whose listing_images[0].url is a non-image URL.

12. Price formatting inconsistent between grid and map; no thousands separator; invalid currency codes echoed verbatim

Problem. formatPrice does plain interpolation (no Intl grouping). The grid uses formatPrice, but the Leaflet pin/popup builds its own string that emits ONLY the bare number for any non-USD currency (no currency suffix), so the same EGP listing shows '3000 EGP' in the grid and '3000' on the map. No thousands separator anywhere, and garbage stored currency codes are rendered verbatim (live data contains currency 'UDDD' and 'NOTACUR', shown as '3200 UDDD' / '100 NOTACUR').

Fix. Replace the two hand-built map price strings with formatPrice(price, currency); upgrade formatPrice to use Intl.NumberFormat for grouping; validate currency against ISO-4217 at listing creation (and/or fall back to a default at render). Clean the garbage 'UDDD'/'NOTACUR' seed rows from the DB.

13. Explore search date inputs lack min and check-out>check-in validation

Problem. The /explore search filter checkIn/checkOut <input type=date> have no min attribute and no relative validation; inverted or past ranges are passed to getListings, which only applies the availability filter when both dates pass a YYYY-MM-DD format check (no order/past check), so an inverted/past range silently behaves like an empty search. Note this is the search-refinement bar only — the actual booking ReservePanel (reserve-panel.tsx:152,168) DOES set min correctly, so the real booking flow is guarded.

Fix. On the explore checkIn input set min=today; on checkOut set min={filters.checkIn||today}; when checkOut<=checkIn, normalize or show an inline hint instead of silently no-opping. Optionally guard order/past in getListings too.
